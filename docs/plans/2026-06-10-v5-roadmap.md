# Roadmap to v5 — samebrain + smartloop

**Status:** Draft 2026-06-10
**Scope:** Four releases (v2–v5) for the samebrain engine and the smartloop skill, versioned in lockstep.

## Why lockstep

smartloop's future depends on data samebrain's hooks must start emitting first.
The dependency spine of this roadmap:

```
v2: emit data      → hooks record sessions; smartloop hardens its trace format
v3: read the data  → local dashboard, evals, fleet status
v4: learn from it  → rule mining, prompt optimization, learned pacing
v5: run everywhere → cross-agent smartloop, portfolio scheduling, self-tuning
```

Each version is independently shippable and useful. No version requires the
next to justify itself.

## Invariants (every version, non-negotiable)

- Zero services: nothing hosted, nothing metered, no daemon.
- Zero LLM API calls in any hot path. Agents run on subscriptions; offline
  optimization (v4) runs on explicit user command only.
- Everything is markdown/JSONL in git. Greppable, editable, portable.
- Engine/overlay split holds: public engine stays generic; consumer repos
  carry private config.
- Merge-only where agents own state; render is idempotent and deterministic.
- Tests in `test/run.mjs` (no framework), CI green on Linux/macOS/Windows.
- Token-frugality is a feature gate: any addition that grows per-session
  context must justify its tokens in the PR description.

## v2 — Widen the pipes (sync everything, start recording)

Goal: every surface synced; every session leaves a trace. ~2–3 weeks of
part-time work.

### samebrain v2

|---|---|---|
| 2.1 | Skills render to Cursor (`~/.cursor/skills/`) and Codex (prompts/commands equivalent) | Same merge-never-delete semantics as Claude target |
| 2.2 | Session telemetry: recall/sync hooks append one JSONL record per session to `telemetry/<machine>/<YYYY-MM>.jsonl` | Fields: ts, agent, session_id, machine, duration_s, cwd_hash. | 2.3 | Telemetry rotation guard | Render warns when current month file >1MB; `--gc` flag rolls files older than N months into per-month summary lines |

| 2.6 | Cursor User Rules drift check: canonical paste-file `global/cursor-user-rules.md`; `render.mjs --check` reminds when its hash changes since last ack | Mitigates the vendor-locked surface; no settings-DB hacking |

|---|---|---|

Goal: the OpenSync value proposition — dashboards, search, eval export —

| # | Deliverable | Notes |
|---|---|---|
| 3.1 | `bin/dashboard.mjs` → static `dashboard.html` (gitignored): sessions/day per agent per machine, smartloop run table, memory index health | No server; open the file. Vanilla JS + inline data, no build step |

### smartloop v3

| # | Deliverable | Notes |

- [x] DeepEval/OpenAI-Evals exports validate against their schemas in a test

- [x] Still zero services: invariants test scans `bin/` + `hooks/` for service/LLM references

## v4 — Close the loop (learning from accumulated data)

Goal: information flows backward — sessions improve the rules, traces improve
the protocol. Gated on data volume, not calendar: start when ~50 smartloop
runs / ~500 sessions have accumulated.

### samebrain v4

| # | Deliverable | Notes |
|---|---|---|
| 4.1 | Rule mining: `/rule-mine` skill scans recent session corrections (user "stop doing X" patterns surfaced from memory + telemetry) and opens a PR editing `global/guardrails.md` | Agent-proposed, human-merged. PR is the safety gate |
| 4.2 | Coordination leases: claims become lease files in git (`coordination/leases/<scope>.lease` with owner + expiry) checked by a render-installed hook | Mechanical enforcement of what `coordination.md` only describes; stale leases expire by timestamp, no daemon |
| 4.3 | Memory decay policy in `/memory-gc`: facts carry last-confirmed dates; unconfirmed facts past N days get flagged for prune | Builds on 2.5 |

### smartloop v4

| # | Deliverable | Notes |
|---|---|---|
| 4.4 | Offline prompt optimization (the design doc's "v2 seam"): `bin/optimize.mjs` exports the trace corpus as a DSPy dataset; an explicit offline run (user's API key, user's command, never automatic) proposes improved contract-compilation text as a PR to `skills/smartloop/SKILL.md` | The one sanctioned LLM-API touchpoint — offline, opt-in, output is reviewable text |
| 4.5 | Learned pacing defaults: `bin/optimize.mjs --pacing` derives per-task-type wait-ladder hints from trace history (e.g. median CI wait per repo) and writes them to a consumer-overlay addendum | Heuristics stay static text in the skill; data only tunes the numbers |
| 4.6 | Regret-driven tier suggestions: when a low-tier verification later produced a failure (reverted commit, reopened criterion), flag the task type for tier-up in the next optimize run | Uses 3.6 labels |

### Exit criteria

- [ ] One rule-mining PR merged with a guardrail improvement traceable to session evidence *(machinery shipped; first PR is data-gated — needs real accumulated corrections)*
- [ ] Optimize run produces a measurable win on a held-out trace set *(exporter + bounded-apply shipped; the optimization run itself is data-gated and user-invoked)*
- [x] Lease check blocks a simulated double-claim in tests

## v5 — Run everywhere, self-tuning

Goal: smartloop is agent-agnostic; the system tunes itself within human-set
bounds. Sequenced after v4 because porting is mechanical while learning is
risky — derisk the hard part first.

### samebrain v5

| # | Deliverable | Notes |
|---|---|---|
| 5.1 | Full agent matrix: Gemini + Copilot memory hooks (when their harnesses support them — tracked as blocked-external until then), opencode/Droid/Pi render targets | Each target lands only with a real harness to test against |
| 5.2 | Dashboard v2: memory-quality and rule-effectiveness panels (which guardrails correlate with fewer corrections) | Still static HTML |

### smartloop v5

| # | Deliverable | Notes |
|---|---|---|
| 5.3 | Cross-agent ports: smartloop on Codex and Cursor — same state-file spec, per-agent liveness shims (each agent's hook/wake primitives mapped in a small adapter table in the skill) | State file was agent-neutral from v1 by design; ports are adapters, not rewrites |
| 5.4 | Portfolio scheduler: `/smartloop portfolio` — priority per run, drain cheap runs while an expensive one is limit-paused, one shared quota model | A skill-level policy over existing park/resume; no daemon |
| 5.5 | Self-tuning tiers: 4.6's suggestions auto-apply within human-set floor/ceiling bounds declared in the consumer overlay | Bounds are the safety contract; engine never lowers a tier below the declared floor |

### Exit criteria

- [ ] The same smartloop run (state file) can be started in Claude Code, parked, and resumed in Codex
- [ ] Portfolio mode drains a 3-item backlog across a simulated limit window in a scripted test
- [ ] A tier auto-adjustment is logged, bounded, and reversible via git

## Sequencing rationale & risks

- **Telemetry before dashboards before learning** — each layer is the next
  one's input; building out of order produces features with no data.
- **v4 is data-gated, not date-gated.** Optimizing on 10 traces overfits;
  the gate (~50 runs) is cheap to wait for because v2/v3 are useful alone.
- **Biggest risk: telemetry token/noise creep.** Mitigation: telemetry is
  hook-side only (zero context cost), and the dashboard is pull-based.
  Nothing in v2–v5 adds per-session context except what the user opts into.
- **Second risk: cross-agent hooks are vendor-moving targets.** Mitigation:
  every target is capability-detected at render time (existing pattern);
  blocked-external items carry an issue, not a deadline.
- **Explicitly never building:** memory servers, vector stores, embeddings
  in the engine, hosted anything, hook-driven iteration (Ralph-style), or
  any LLM call the user didn't explicitly invoke.

## Working agreement

- Each numbered deliverable = one GitHub issue (enckequity/samebrain),
  claimed per `coordination.md`, branch under the agent namespace, PR-only.
- Design docs precede code for 3.5, 4.1, 4.4, 5.3 (the four with real
  design risk); the rest go straight to TDD against `test/run.mjs`.
- Conventional commits; README updated in the same PR as any user-facing
  change.
